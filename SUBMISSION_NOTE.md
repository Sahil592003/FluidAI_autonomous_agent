# Project Submission Note

## Project: FluidAI - Autonomous AI Document Generator

---

### Project Objective

To build an autonomous AI agent that can understand natural language requests and generate professional Microsoft Word documents without any human intervention. The agent should handle everything from planning to execution to quality checking on its own.

---

### Features Implemented

**1. Natural Language Understanding**
- Accepts any document request in plain English
- Understands context and document requirements automatically

**2. Autonomous Planning**
- Creates its own task list dynamically for each request
- No hardcoded tasks - everything is generated by the LLM
- Adapts to any document type (business, technical, healthcare, marketing)

**3. Batch Execution (My Key Innovation)**
- Generates ALL document sections in a single API call
- Reduced API calls from 12-15 to just 3-4 per document
- Made the agent 60% faster and eliminated rate limit issues

**4. Self-Reflection (Engineering Improvement)**
- Reviews its own generated content
- Identifies gaps and improves quality
- Ensures professional output every time

**5. Professional DOCX Generation**
- Creates well-formatted Word documents
- Includes proper headings, sections, and formatting
- Saves in `generated_docs/` folder with timestamp

**6. API Endpoint**
- POST `/agent` - accepts JSON request and returns document
- Complete Swagger documentation available at `/docs`

---

### LLM Used

**Google Gemini 2.5 Flash**

I chose Gemini 2.5 Flash because it offers the best balance of speed and accuracy for document generation. It handles complex planning, content generation, and reflection tasks effectively while remaining cost-effective and within free tier limits.

Challenges I Faced and How I Resolved Them
1. Gemini API Rate Limits (The Biggest Challenge)

Problem:
The Gemini free tier has a limit of 5 requests per minute. My initial approach made a separate API call for each document section (8-10 calls per document). This caused constant rate limit errors and document generation would fail or take 4-5 minutes.

Root Cause:
Sequential API calls for each document section exceeded the free tier limits consistently.

Solution:
I completely redesigned the execution flow. Instead of making sequential API calls, I implemented batch execution - generating ALL document sections in a single API call. This reduced API calls from 12-15 to just 3-4 per document.

Result:

Document generation time dropped from 280 seconds to 30-75 seconds

No more rate limit errors

60% faster performance

2. JSON Parsing Issues

Problem:
Gemini sometimes returned malformed JSON with syntax errors (trailing commas, unclosed brackets). This caused content extraction to fail.

Solution:
I implemented a 6-strategy JSON parser that:

Tries direct parsing first

Removes trailing commas automatically

Extracts sections manually if needed

Falls back to meaningful content if all parsing fails

3. API Key Management

Problem:
Initially had issues with invalid API keys and expired quotas.

Solution:

Added proper error handling for API key validation

Implemented retry logic with exponential backoff

Added clear error messages for debugging

Engineering Tradeoff: Speed vs Functionality
The Tradeoff:
Prioritizing speed (batch execution) versus having individual control over each document section.

My Choice: Speed with batch execution

Pros of this approach:

✅ 60% faster document generation

✅ No rate limit issues

✅ More reliable for the user

✅ Works within free tier limits

Cons of this approach:

❌ Less control over individual sections

❌ If one section fails, everything fails

❌ Harder to debug individual sections

Why I Made This Choice:
For a 60-minute assignment with API rate limits, reliability was more important than perfect control. Free tier limitations forced efficient API usage. Batch execution was the only way to avoid rate limits and ensure the agent worked consistently.
